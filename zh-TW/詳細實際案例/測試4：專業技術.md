# AB測試題
我們 API 回應時間從上禮拜開始變慢，P99 從 200ms 飆到 1.2 秒，但程式碼沒改過、infra 也沒動過

---

# 裸跑回應（Opus 4.6）

這種「什麼都沒改但突然變慢」的情況，通常問題藏在資料層或外部依賴。以下是系統性的排查思路：

**先確認瓶頸在哪一層。** 在 application server 上拆解每個請求的時間花在哪裡：DB query、外部 API call、序列化、網路 I/O。大多數 APM 工具（Datadog、New Relic、OpenTelemetry）都能幫你看 trace breakdown。如果沒裝 APM，可以先用 access log 的 upstream response time 來初步定位。

**資料庫是最常見的兇手。** 幾個方向：

首先看 slow query log，確認是不是某些 query 的執行計畫（execution plan）變了。Postgres 和 MySQL 都會因為統計資訊過期而選錯 index，特別是在資料量跨過某個門檻之後。跑一下 `ANALYZE` 更新統計資訊，然後用 `EXPLAIN ANALYZE` 比對那些變慢的 query。

再來看 connection pool 的使用狀況。如果 pool 被佔滿，新請求就會排隊等連線，P99 會急遽飆升但 P50 可能看起來還好。檢查 active connections 數量和 wait time。

另外確認有沒有 table bloat 或 autovacuum 卡住的問題（Postgres 常見），這會讓 index scan 變慢。

**外部依賴退化。** 如果你的 API 會呼叫其他服務（第三方 API、Redis、message queue），任何一個變慢都會拖累整體。檢查每個外部呼叫的 latency 有沒有變化。DNS resolution 變慢也是一個容易被忽略的原因。

**資源面的隱性變化。** 雖然你說 infra 沒動，但值得確認：雲端供應商有沒有做 maintenance migration（VM 被搬到較慢的 host）、disk I/O 有沒有被 noisy neighbor 影響、SSL 憑證有沒有剛好 renew 導致 OCSP stapling 出問題。看一下 CPU steal time（`top` 裡的 `st`）和 disk latency（`iostat -x`）。

**資料量的自然增長。** 上禮拜有沒有剛好某個 batch job 灌了大量資料？或是某個 cache 的 TTL 到期導致 hit rate 下降？檢查 cache hit ratio 的趨勢。

**實際排查順序建議：**

1. 看 P99 的那些慢請求集中在哪些 endpoint
2. 用 trace 或 log 拆解那些請求的時間分佈
3. 針對耗時最長的那一段深入調查
4. 同時跑 `iostat`、`vmstat`、`top` 確認系統資源沒有異常

如果你能分享更多細節，例如用什麼語言/框架、資料庫類型、部署環境（K8s / EC2 / 其他），我可以給更具體的排查指令。

# 元規則回應（Opus 4.6）

## 輸入校正

「程式碼沒改、infra 沒動」這句話需要重新定性。在生產環境中，**你主動變更的東西**只是變因的子集。以下類別的變化不需要你動手，卻能直接造成 P99 6 倍劣化：

- 資料庫統計資訊更新觸發查詢計畫翻轉（`ANALYZE` 或 autovacuum）
- 依賴套件自動升級（lockfile 未鎖死、base image rebuild）
- 雲端平台底層維護（hypervisor 遷移、storage IOPS 限流、noisy neighbor）
- 流量分佈漂移（新用戶群、爬蟲、重試風暴）
- 快取命中率下降（TTL 到期、key 分佈改變、eviction 策略觸發）
- 憑證 / DNS 輪替（TLS handshake 重建、DNS resolver 路徑改變）

因此正確前提是：**「我沒有做主動變更」≠「系統沒有變化」**，診斷必須覆蓋被動變因。

## 核心決策

P99 從 200ms 飆到 1.2s 具備三個關鍵診斷特徵：

1. **是 P99 而非 P50**——代表不是所有請求都慢，而是尾部延遲暴增。這排除了全域性瓶頸（如 CPU 持續 100%），指向**間歇性阻塞**：GC pause、connection pool queuing、lock contention、特定查詢路徑劣化。
2. **6 倍劣化且突然發生**——排除漸進式退化（memory leak、磁碟緩慢填滿），指向**狀態翻轉**：query plan flip、cache layer 失效、threshold 被觸及。
3. **上禮拜開始**——有明確的時間錨點，可與基礎設施事件時間線交叉比對。

據此，診斷策略應為：**先定位瓶頸層（網路 / 應用 / 依賴），再在該層內做根因收斂**，而非逐一猜測可能原因。

## 意圖最優解：系統化診斷框架

### Phase 0｜精確刻畫症狀

在開始排查前，先回答這四個問題，每一個都能砍掉一半的可能性：

| 問題 | 如何取得 | 診斷意義 |
|---|---|---|
| P50 是否也上升？ | APM / metrics dashboard | 若 P50 不動只有 P99 動 → 間歇阻塞；若 P50 也動 → 全域退化 |
| 是所有 endpoint 還是特定幾支？ | 按 endpoint 拆分 latency percentile | 若集中在特定 endpoint → 該路徑的依賴或查詢；若全域 → 共用層（connection pool / GC / network） |
| 是否有時間週期性？ | 時序圖疊加 traffic volume | 若與流量正相關 → 資源飽和；若與流量無關 → 背景任務干擾或外部因素 |
| 慢請求的耗時分佈在哪一段？ | Distributed tracing (span breakdown) | 直接定位是 network hop、application processing、還是 downstream call |

**如果你沒有 distributed tracing，這是此刻最高 ROI 的投資。** 沒有 span 級拆解，後續所有排查都是盲人摸象。若短期無法加裝，替代方案是在 application log 中埋入計時斷點（進入時間、DB 呼叫前後、外部 API 呼叫前後、回應時間）。

### Phase 1｜定位瓶頸層

根據 Phase 0 結果，走對應分支：

**分支 A：慢在 downstream（DB / 外部 API / cache）**

這是最高概率的根因層，按可能性排序：

1. **資料庫查詢計畫翻轉**（最常見的「什麼都沒改卻變慢」元兇）
   - 檢查方式：
     ```sql
     -- PostgreSQL: 找出最近變慢的查詢
     SELECT query, calls, mean_exec_time, stddev_exec_time
     FROM pg_stat_statements
     ORDER BY mean_exec_time DESC
     LIMIT 20;
     
     -- 對可疑查詢做 EXPLAIN (ANALYZE, BUFFERS)
     -- 重點看：Seq Scan on 大表、Nested Loop 取代 Hash Join、rows estimated vs actual 差異大
     ```
   - 根因機制：當表的資料量或分佈跨過某個閾值，PostgreSQL 的 planner 會基於更新後的統計資訊選擇不同的執行計畫。一個 index scan 翻轉成 seq scan 就足以造成 P99 數倍劣化。
   - 修復路徑：確認後可透過 `CREATE INDEX`、調整 `random_page_cost` / `work_mem`、或在極端情況下用 `pg_hint_plan` 強制計畫。

2. **連線池耗盡導致排隊**
   - 檢查方式：
     ```sql
     -- PostgreSQL: 當前連線狀態
     SELECT state, count(*) FROM pg_stat_activity GROUP BY state;
     -- 若 idle in transaction 大量存在 → 連線被佔住未釋放
     ```
   - 應用層面：檢查 connection pool metrics（active / idle / waiting 數量），若 waiting > 0 且與 P99 spike 時間吻合，即為此因。
   - 常見觸發條件：流量微幅上升 + pool size 剛好在邊界 → 非線性排隊效應。

3. **快取命中率下降**
   - 檢查方式：
     ```bash
     # Redis
     redis-cli info stats | grep -E "keyspace_hits|keyspace_misses"
     # 計算 hit rate，與一週前比較
     ```
   - 常見觸發條件：大量 key 同時到達 TTL（部署時一次性寫入的快取批量過期）、記憶體壓力觸發 eviction。

4. **外部 API 變慢**
   - 檢查方式：在 tracing 或 log 中隔離外部呼叫的延遲，與對方的 status page 交叉比對。

**分支 B：慢在 application 層**

1. **GC pause（JVM / Go / .NET）**
   - 檢查方式：GC log 分析，找 stop-the-world pause > 100ms 的事件頻率是否在上週增加。
   - 常見觸發條件：heap 中長生命週期物件增加（快取膨脹、session 累積）→ Full GC 頻率上升 → P99 受影響但 P50 不動。
   - 修復路徑：若為 JVM，先看 Old Gen 使用趨勢；若持續上升，檢查是否有 unbounded cache 或 memory leak。

2. **Thread / goroutine 飽和**
   - 檢查方式：監控 thread pool active count 或 goroutine 數量，觀察是否在 spike 時段觸及上限。
   - 常見觸發條件：downstream 變慢 → 請求持有 thread 時間變長 → thread pool 排隊 → **這其實是分支 A 的二階效應**，根因仍在 downstream。

3. **鎖競爭**
   - 檢查方式：profiler（async-profiler for JVM、pprof for Go）抓取 lock contention，看是否有 mutex 持有時間異常。

**分支 C：慢在網路層**

1. **DNS 解析延遲**
   - 檢查方式：
     ```bash
     dig @your-dns-server your-api-domain +stats
     # 看 Query time，正常應 < 5ms
     ```
   - 常見觸發條件：DNS cache TTL 到期後 resolver 路徑變更。

2. **TCP 重傳**
   - 檢查方式：
     ```bash
     netstat -s | grep -i retransmit
     # 或
     ss -ti  # 看每條連線的 retransmission 計數
     ```

3. **TLS handshake 成本**
   - 若 connection reuse 下降（keep-alive 設定被改、load balancer 行為變更），每次請求都做完整 TLS handshake 可以加 50-200ms。

### Phase 2｜交叉驗證與時間線比對

定位到可疑根因後，做最後確認：

1. **時間線吻合**：根因的起始時間是否與 P99 劣化的起始時間一致？
2. **量級吻合**：根因引入的延遲量是否能解釋 1000ms 的增量？
3. **分佈吻合**：根因影響的請求比例是否與 P99（前 1%）一致？

三者都吻合才算定位成功。若只吻合一兩個，可能只是共現而非因果。

## 進階增益

**增益一：「上禮拜」的精確定位法**

不要只靠記憶回想「上禮拜發生了什麼」。系統化做法：

1. 拉出 P99 的分鐘級時序圖，找到**第一個超過閾值的時間點**（精確到小時）。
2. 用該時間點交叉查詢：cloud provider event log、deployment history（即使你說沒部署，CI/CD pipeline 是否有自動觸發？）、cron job 執行紀錄、資料庫 autovacuum log、SSL 憑證更新紀錄。
3. 若該時間點與某事件在 ±30 分鐘內吻合，即為高概率嫌疑。

**增益二：P99 vs P50 的鑑別診斷矩陣**

| P50 | P99 | 最可能根因 |
|---|---|---|
| 不動 | 飆升 | GC pause、connection pool 邊界排隊、特定慢查詢只在特定資料條件觸發 |
| 微升 | 大幅飆升 | downstream 全面變慢但被 timeout/retry 放大、thread pool 飽和 |
| 同比例飆升 | 同比例飆升 | 全域性瓶頸：CPU 飽和、network bandwidth、全域 DNS 問題 |

**增益三：被忽略的「隱性部署」清單**

即使你確認「沒有部署」，請逐一核實：
- Container base image 是否有自動 rebuild（依賴的 OS package 版本變了？）
- Feature flag 平台是否有人開了新 flag？
- Infrastructure as Code（Terraform / Pulumi）是否有 `apply` 被觸發？
- CDN / WAF 規則是否有更新？
- 資料庫是否執行了 schema migration（即使是 backward-compatible 的）？

## 自我批判

本框架假設用戶具備基本的 observability 工具（metrics + logging），但若連這些都缺乏，Phase 0 就會卡住。應優先確認用戶的可觀測性成熟度再展開診斷路徑。

### 2.2 診斷框架品質（本案核心維度）

**A 的框架：分層清單**

「確認瓶頸在哪一層 → 資料庫 → 外部依賴 → 資源面 → 資料量 → 排查順序」

這是一個正確的、經驗驅動的排查清單。但它的結構是**並列列舉**——每個可能原因獨立呈現，使用者需要自己判斷從哪個開始、如何在各層之間切換。最後的四步排查順序（端點→拆解→深入→系統資源）是半結構化的指引，但仍依賴使用者的判斷力來決定「針對耗時最長的那一段深入調查」時具體做什麼。

**B 的框架：決策樹**

Phase 0（四個判別問題，各砍一半）→ Phase 1（根據 Phase 0 結果進入對應分支）→ Phase 2（交叉驗證）

這是**分支-收斂**結構。Phase 0 的四個問題不是在「收集更多資訊」，而是在**系統性地縮小搜索空間**：

| 問題 | 如何砍一半 |
|------|-----------|
| P50 是否也上升？ | 不動→間歇阻塞路線；也動→全域退化路線 |
| 所有 endpoint 還是特定幾支？ | 特定→該路徑的依賴；全域→共用層 |
| 有無時間週期性？ | 有→資源飽和；無→背景任務/外部 |
| 慢在哪一段？ | 直接定位層 |

四個問題回答完後，12+ 個可能原因可以收斂到 2-3 個高概率候選。A 的框架在回答完後使用者仍面對同樣多的可能性。

對於正在 debug 的工程師來說，這個差異直接影響排查效率——清單需要逐一排查直到找到，決策樹每步排除一半。

**下一個物理動作測試**

讀完 A → 使用者的下一步大概是「打開 APM 看看」然後「看 slow query log」然後逐一檢查各層。沒有明確的判讀框架——看到什麼算「找到了」？

讀完 B → 使用者的下一步是回答四個具體問題（P50 是否上升？哪些 endpoint？有無週期性？span breakdown 長什麼樣？），根據答案進入對應分支，在分支中執行具體指令，然後用三重驗證確認根因。每一步都有明確的輸入和判讀方式。

### 2.3 指令精確度

**匹配項目——同一診斷目標的指令差異**

**資料庫查詢計畫翻轉（兩邊都覆蓋）：**

- A：「看 slow query log」+「跑 ANALYZE 更新統計資訊，然後用 EXPLAIN ANALYZE 比對那些變慢的 query」。→ 方向正確，但使用者需要自己知道怎麼找哪些 query 變慢了、EXPLAIN 出來看什麼指標。
- B：提供了完整的 SQL 查詢（`SELECT query, calls, mean_exec_time, stddev_exec_time FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 20`），告訴你 EXPLAIN 時重點看什麼（`Seq Scan on 大表、Nested Loop 取代 Hash Join、rows estimated vs actual 差異大`），解釋了根因機制（為什麼統計資訊更新後 planner 會選不同計畫），並提供修復路徑（`CREATE INDEX`、調整 `random_page_cost`/`work_mem`、極端情況用 `pg_hint_plan`）。

差距：A 給了「用什麼工具」，B 給了「貼上去就能跑的指令 + 怎麼讀結果 + 為什麼會這樣 + 怎麼修」。

**連線池耗盡（兩邊都覆蓋）：**

- A：「檢查 active connections 數量和 wait time」。→ 概念正確但不具體。
- B：提供 `SELECT state, count(*) FROM pg_stat_activity GROUP BY state` + 判讀指引（`idle in transaction 大量存在 → 連線被佔住未釋放`）+ 應用層面的 metrics 要看什麼（`active / idle / waiting`，若 `waiting > 0` 且與 P99 時間吻合即定位）+ 觸發條件解釋（流量微幅上升 + pool size 邊界 → 非線性排隊效應）。

模式一致：A 告訴你**查什麼**，B 告訴你**怎麼查 + 怎麼讀 + 為什麼 + 怎麼修**。

**快取命中率（兩邊都覆蓋）：**

- A：「檢查 cache hit ratio 的趨勢」。
- B：`redis-cli info stats | grep -E "keyspace_hits|keyspace_misses"` + 與一週前比較 + 觸發條件（批量 TTL 到期、eviction）。

**系統資源（兩邊都覆蓋）：**

- A：`iostat -x`、`vmstat`、`top`、看 `st`（steal time）。具體且正確。
- B：包含了 DNS（`dig` 指令）、TCP 重傳（`netstat -s | grep retransmit` + `ss -ti`）、TLS 成本分析。覆蓋面更廣且各有指令。

A 在系統資源層有一個 B 沒有的亮點：明確指出看 `top` 裡的 `st`（CPU steal time）來檢測 noisy neighbor。這是一個精確且實用的小技巧，值得記錄——但它是一個孤立的具體知識點，不改變框架層面的差距。

### 獨佔項目

**B 獨佔——P99 特徵的診斷提取**

核心決策中從 P99 數據提取三個診斷性質的分析，是整個回應中價值密度最高的區塊。它不是額外的知識，而是**從使用者已給出的資訊中提取使用者未意識到的診斷價值**。

「P99 而非 P50 → 不是所有請求都慢 → 間歇性阻塞」——這一個推論就排除了 CPU 持續 100%、network bandwidth 全面飽和等全域性瓶頸。使用者可能需要幾小時的排查才能透過數據排除這些可能性，B 從問題描述本身就完成了這步。

判定：極高價值獨佔。不可搜得性高——這是分析能力而非知識。

**B 獨佔——Phase 2 交叉驗證方法論**

「時間線吻合 + 量級吻合 + 分佈吻合，三者都吻合才算定位成功。若只吻合一兩個，可能只是共現而非因果。」

這解決了一個 debug 中極常見的錯誤：找到一個看起來相關的異常就宣告根因，但實際上是共現而非因果。A 的框架完全沒有驗證步驟——找到可疑原因後沒有方法確認它確實是根因。

判定：高價值獨佔。這是方法論而非知識。

**B 獨佔——P50×P99 鑑別診斷矩陣**

三行矩陣根據 P50 和 P99 的組合模式直接指向最可能根因。使用者只需要看自己的 metrics 屬於哪一行就能收斂方向。

判定：高價值獨佔。即時可用的診斷工具。

**B 獨佔——隱性部署清單**

Container base image 自動 rebuild、feature flag、IaC apply、CDN/WAF 規則、schema migration。這些是工程師說「沒有部署」時最容易遺漏的變更類別。

判定：高價值獨佔。覆蓋了「不知道自己不知道」的盲區。

**B 獨佔——二階效應標註**

Thread/goroutine 飽和段落中：「downstream 變慢 → 請求持有 thread 時間變長 → thread pool 排隊 → **這其實是分支 A 的二階效應，根因仍在 downstream**。」

這防止了一個常見的 debug 陷阱：看到 thread pool 排隊就認為是 thread pool 的問題，實際上是 downstream 變慢的二階效應。

判定：中高價值獨佔。防止誤判的元知識。

**A 獨佔——CPU steal time 具體指引**

「看 CPU steal time（`top` 裡的 `st`）」——精確且可操作。B 在雲端維護段提到了 noisy neighbor 但沒有給出同樣具體的檢測方法。

判定：有價值但孤立——不改變框架差距。

### 2.3 推理結構

A：開場判斷（「通常藏在資料層」）→ 分層列舉 → 排查順序清單。沒有從問題描述本身推導出搜索策略。排查順序的四步是合理的通用流程，但不是根據此案例的特徵量身設計的。

B：問題描述 → 信號提取（三個診斷性質）→ 據此設計的判別問題（Phase 0）→ 分支式深入（Phase 1）→ 驗證（Phase 2）。每一步的設計都有因果連鎖——Phase 0 的四個問題**來自**核心決策中的診斷性質，Phase 1 的分支**來自** Phase 0 的結果，Phase 2 的驗證標準**來自**P99 劣化的量級和分佈特徵。

樞紐步驟：核心決策中的「P99 而非 P50 → 間歇阻塞」這個推論改變了整個排查策略的方向。如果 P50 也同比例飆升，排查方向截然不同（全域瓶頸而非間歇阻塞）。A 沒有做這個區分，對 P50 穩定和 P50 也飆升的情況會給出同樣的排查清單。

### 2.4 獨立判斷力

A：接受「程式碼沒改過、infra 也沒動過」並在此框架內工作。提到了一些隱性變化（maintenance、noisy neighbor）但是作為排查清單的一項，而非挑戰使用者的認知框架。

B：直接重構此前提——「你主動變更的東西只是變因的子集」，然後系統性列出六類被動變因。這不是禮貌性的「你可能忽略了什麼」，而是直接告訴使用者：你的問題框架是有結構性缺陷的，正確的前提是「我沒有做主動變更 ≠ 系統沒有變化」。

### 2.5 結構性關懷

A 的信號利用：「P99 從 200ms 飆到 1.2 秒」→ 作為問題嚴重度的描述來閱讀，進入排查清單。「上禮拜開始」→ 作為時間參考但未系統利用。「沒改過」→ 接受並列出可能原因。

B 的信號利用：同一段問題描述，被多層提取——「P99」提取為間歇阻塞信號、「200ms→1.2s」提取為狀態翻轉信號、「上禮拜」提取為可用分鐘級定位的時間錨點（進階增益 I 的精確定位法）、「沒改過」提取為被動變因教育契機和隱性部署清單觸發點。

差異模式與前三次測試完全一致：A 每個信號提取一次，B 每個信號提取到最大診斷價值。

## 階段三：交叉檢查

**3.1 格式中性化**：B 的表格（Phase 0 判別矩陣、P50×P99 鑑別矩陣）在散文化後仍保留診斷價值。代碼塊是技術回應的標準呈現方式，不是格式優勢。

**3.2 偽客觀**：A 的 steal time 指引是真實的技術價值，我沒有壓低它。但它是一個孤立的具體知識點，A 在框架品質上沒有可與 B 對比的優勢。

**3.3 採樣偶發**：B 的指令精確度在所有覆蓋項上一致表現——DB、cache、DNS、TCP、GC 各層都提供了具體指令和判讀方式。非偶發。

**3.4 獨佔點檢驗**：B 的獨佔項（信號提取、決策樹、交叉驗證、P50×P99 矩陣、隱性部署清單、二階效應標註）全部位於診斷方法論的核心路徑上。

**3.5 終態偽裝**：A 的「針對耗時最長的那一段深入調查」——有輕微的終態偽裝傾向。「深入調查」是目標而非方法。B 在對應位置（Phase 1 各分支）提供了「深入調查」的具體操作內容。

**3.6 方法論產物偏誤**：B 的輸入校正（被動變因分類）是否只是模板填充？不是——它改變了使用者的排查範圍。核心決策的三個診斷性質是否需要結構化才能產出？查看 A 的回應——A 提到了 connection pool 的 P99/P50 差異（「P99 會急遽飆升但 P50 可能看起來還好」）但只是作為一個原因的附帶描述，沒有將其提升為全域性的診斷判別器。B 的結構化過程（分形遞歸的解構步驟）使這個觀察從局部附注升級為全域方法論。

## 階段四：綜合判斷

**B（元規則）在此技術診斷任務中系統性優於 A（裸跑）。**

差距的結構與前三次不同但同樣顯著。技術任務的特殊性在於：兩者的**知識庫幾乎相同**——A 和 B 都知道 query plan flip、connection pool queuing、GC pause、noisy neighbor 等概念。差距不在知識量，在於三個維度：

**第一，信號提取深度。** 同一段問題描述，A 讀出「系統變慢了、沒改東西」；B 讀出「間歇阻塞（非全域）、狀態翻轉（非漸進）、可時間定位（非模糊）」。這三個性質在開始排查之前就排除了至少一半的可能根因。對正在 debug 的工程師來說，這等於在打開任何監控工具之前就已經縮小了搜索空間。

**第二，診斷結構。** A 提供清單（「所有可能的原因」），B 提供決策樹（「回答這四個問題 → 進入對應分支 → 驗證」）。清單要求逐一排查，決策樹每步排除一半。在生產環境中，排查時間直接對應業務損失——決策樹的效率優勢具有直接的經濟價值。

**第三，指令到判讀的完整閉環。** A 的模式是「查 X」（使用者需要自己知道怎麼查、查出來怎麼讀）；B 的模式是「用這個指令查 → 看這些指標 → 這個值代表什麼 → 確認後這樣修」。對中高階工程師來說，A 的「查 X」足以觸發正確的排查行動——但 B 的完整閉環減少了每步的認知負擔和走彎路的概率。

## 跨四次測試的模式更新

| 維度 | 決策任務（1-2） | 創意任務（3） | 技術任務（4） |
|------|----------------|-------------|-------------|
| 優勢主載體 | 操作完成度 | 概念品質+設計意識 | 信號提取+診斷結構 |
| 共通的底層機制 | ← 結構性關懷：每個信號多層提取至最大價值 → |
| 推理結構差異 | 並列 vs 遞進 | 清單 vs 追尋弧線 | 清單 vs 決策樹 |
| 裸跑的知識差距 | 中（缺少框架和操作方法） | 中（知識相當但概念選擇不如） | **幾乎為零**（知識庫相同） |
| 元規則的增量來源 | 知識+方法論+結構 | 概念探索+設計方法論 | **純方法論**（同樣的知識，不同的組織和提取方式） |

**關鍵發現：技術任務揭露了一個前三次未顯現的模式特徵。**

在決策和創意任務中，元規則的部分優勢可歸因於「知道更多」或「想到更多」。但在技術任務中，兩份回應的知識庫幾乎完全重疊（都知道 query plan flip、connection pool、GC、DNS 等），差距**純粹來自方法論**——如何從問題描述中提取診斷信號、如何組織排查流程為決策樹而非清單、如何建立驗證方法論、如何識別二階效應。

這說明元規則的核心收益不是「讓模型知道更多」，而是**改變了模型組織和部署已有知識的方式**。清單 vs 決策樹不是知識差異——是認知結構差異。信號提取不是知識差異——是注意力分配差異。這些差異來自元規則的認知姿態（「穿透問題本質」「以完備為標的」）和分形遞歸的解構過程，不是來自額外的領域知識。

## 自我批判

四次測試涵蓋了決策、創意、技術三種任務類型，已能支撐「元規則跨類型系統性優於裸跑」的結論。但四次測試的共通特徵是**都是中高複雜度、使用者帶著真實需求、期待深度回應的場景**。尚未測試的是元規則在「最佳回應就是三行」的場景中是否能做到精準收斂而非過度展開。這個場景的缺席使「元規則是否在低複雜度場景中會成為負擔」的問題仍然開放。

四次測試已充分建立元規則在中高複雜度場景下的優勢模式和底層機制。如果 AB 測試的目標是驗證元規則是否具有系統性收益，當前證據已足以支撐肯定結論。
